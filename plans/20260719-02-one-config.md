# One Config: the Same Document for `ewm one`

- **Design doc:** `notes/JSON_CONFIG.md` (the document model, sources,
  merge rules, and validation split — this plan extends that machinery to
  the Apple 1 / Replica 1 without changing its architecture). Update it
  with *as built* notes as phases land.
- **Status:** draft — iterate here before kickoff
- **Target:** `main`; PR granularity decided at kickoff

## Goal

`ewm one` speaks the same configuration language as `ewm two`:

```
ewm one --config builtin:replica1 \
        --config-overlay basic-in-rom.json \
        --set cpu:strict=true
ewm one --print-config
```

One document type describes *any* EWM machine — `machine.model` says
which — and the four sources, builtins, `--set`, and `--print-config`
behave identically across subcommands. The Apple 1 has no slots or
cards, but the configurable bits it does have all map onto existing
keys:

| `one` today | Config key |
|---|---|
| `--model apple1\|replica1` | `machine.model` |
| `--memory ram\|rom:addr:path` (decimal only) | `machine.memory` (hex *or* decimal, per region) |
| `--trace [file]` / `--trace=` | `debug.trace` |
| `--strict` | `cpu.strict` |

## Why one document (and not a second schema)

A config file should fully describe a machine, including *which*
machine — that is what makes `.ewmachine` bundles, the bootloader, and
a future `ewm --config x.json` (subcommand picked by model) possible.
So `Model` grows two tokens rather than `one` growing a parallel config
system:

- `Model` = `2plus | 2e | apple1 | replica1`, with a **family**
  (`apple2` vs `apple1`) derived from the token.
- **Validation stays split** exactly as C2 built it: structure per
  fragment, completeness on the final document. The *family
  cross-checks* join the completeness pass, next to "aux is //e-only":
  - `machine.slots` / `machine.aux` — apple2-family only;
  - `display.*`, `cpu.speed`, `input`, `boot`, `remote`, `state` —
    apple2-family only *for now* (the Apple 1's TTY is fixed green,
    its clock is fixed 1.023 MHz, and remote/state for `one` are
    recorded backlog: REMOTE.md Phase 7, STATE.md). **Rejected, not
    ignored** — a config that says `display` on an `apple1` machine is
    wrong, and silence would hide it. Each error names the offending
    key and the family.
  - `cpu.strict`, `debug.trace`, `machine.memory` — valid everywhere.
- **Subcommands enforce family**: `ewm two --config builtin:replica1`
  errors with `machine.model: "replica1" is an ewm one machine (run:
  ewm one --config …)`, and vice versa. (A unified launcher that picks
  the subcommand from the model is backlog, not this plan.)

## Phases

| Phase | Description | Size | Status |
|---|---|---|---|
| O1 | `Model` tokens + family cross-checks; `two` rejects one-family models; both schemas regenerate | M | Not started |
| O2 | Built-ins: `configs/apple1.json`, `configs/replica1.json`, embedded + self-containment sweep | S | Not started |
| O3 | `one` grows the sources: shared pass-1 loop, `apply_config`/`options_to_config` for `one`, `--print-config` | M/L | Not started |
| O4 | Retire `one`'s flags (`--model`, `--memory`, `--trace`, `--strict`); boo/main.rs sweep | S/M | Not started |
| O5 | Docs: README `one` profile entries, JSON_CONFIG as-built, REMOTE Phase 7 note | S | Not started |

O1 → O2 → O3 in order (each builds on the previous); O4 needs O3; O5
last. Every phase: standard gates (fmt, clippy `-D warnings`, full
`cargo test` incl. golden-BMP), `readme_two_examples_parse`, plus the
phase gates below.

### O1 — Model tokens and family cross-checks

- `config::Model` gains `Apple1` / `Replica1` (serde tokens `apple1`,
  `replica1`) and a `family()` accessor; `Model::two_type()` becomes
  family-scoped (panic-free: callers check family first).
- Completeness pass (`validate_complete`) gains the family table above.
  Existing two-family behavior is untouched — the whole suite is the
  tripwire.
- `two::parse_options` rejects one-family models with the
  cross-subcommand message; unit tests pin both directions.
- Both schemas regenerate (the `Model` enum enumeration changes; no
  structural schema change).
- **Gate:** cross-check unit tests (slots on apple1, display on
  replica1, valid minimal one-configs); the cross-subcommand errors;
  schema golden test.

### O2 — Built-in configs for `one`

- `configs/apple1.json` — "Classic Apple 1 — 6502, 8KB RAM, Woz
  Monitor"; `configs/replica1.json` — "Replica 1 — 65C02, 32KB RAM,
  KRUSADER" (32K per the code and README; the `one --help` text that
  says 48KB is a drift, fixed here). Both are minimal: `description` +
  `machine.model` — the ROMs are baked into the machine, not config.
- Naming keeps the convention: name = model token = file stem.
- `builtin:list` grows to four names. **Decision at kickoff**: list all
  four everywhere (recommended — the description says which machine
  family; simpler) or filter per subcommand. Error strings that pin
  `(available: 2e, 2plus)` churn either way.
- **Gate:** self-containment sweep covers the new files;
  `builtin_names_match_their_model` extended; `builtin:list` output.

### O3 — `one` speaks the sources

- Extract `two`'s pass-1 source loop (config/overlay/set collection,
  strict command-line order, default-document seeding, `builtin:list`
  handling) into a shared helper — parameterized by the **default
  machine seed** (`two`: `{"machine":{"model":"2plus"}}`; `one`:
  `{"machine":{"model":"replica1"}}`, matching today's default — the
  `--help` text claiming `apple1` is the default is a drift, fixed
  here) and by the **slots-materialization hook** (supplied by `two`,
  absent for `one`; a one-family document never materializes a ][+
  slot table).
- `one::main` uses it: `--config`, `--config-overlay`, `--set`,
  `--print-config` all work; `from_document` + a small
  `apply_config`/`options_to_config` pair for `one`'s options (model,
  memory, trace, strict). `--print-config` reuses `compact_document`.
- `two`'s behavior must be bit-identical after the refactor (suite +
  README check are the tripwire).
- **Gate:** `one` option tests mirroring `two`'s (sources compose in
  order; overlay-only extends the default machine; `--print-config`
  round-trips through `--config`; family errors); an integration test
  booting the Woz Monitor via `--config builtin:apple1` (the
  `one_boot.rs` harness exists).

### O4 — Retire `one`'s flags

- `--model`, `--memory`, `--trace`/`--trace=`, `--strict` removed
  outright — same v0.0.1 ruling as plan 20260719-01; retired flags are
  unknown flags. `one`'s private `parse_memory_option` and
  `MemoryOption` go with them.
- Sweep the callers: `main.rs` `run_boo` boots the Apple 1 / Replica 1
  via `--set machine:model=…` (or `--config builtin:…` — pick at
  kickoff); README `one` examples; `REMOTE.md` Phase 7 example
  (`one --model replica1 --serve …` → config spelling).
- **Gate:** retired-flag test; boo menu still boots all three machines
  (existing behavior tests); README check.

### O5 — Docs

- README: `one` gets its profiles alongside the `two` section
  (builtin:apple1 / builtin:replica1, JSON + boot commands); Running
  examples use the config spellings.
- `notes/JSON_CONFIG.md`: as-built section (family cross-checks, the
  one-document decision, builtins); schema inventory notes which keys
  are apple2-family-only.
- `notes/REMOTE.md` Phase 7: note that the config surface it wanted now
  exists — serving `one` needs only the `remote` block allowed for the
  family.
- **Gate:** README examples run verbatim (the existing scripted check —
  extend the extractor to `-- one` commands); standard gates.

## Hazards

- **The shared pass-1 refactor** touches `two`'s most-exercised code
  path; land it inside O3 with zero behavior change, proven by the
  untouched suite.
- **Materialization leakage**: the default-slots rule must never fire
  for a one-family document — the hook parameterization plus a test
  (one-overlay carrying `machine.slots` errors in validation, not
  after materialization).
- **Error-string churn**: `(available: 2e, 2plus)` is pinned in several
  tests; O2 updates them once.
- **`Model::two_type()`** is called in `two`-only paths today; O1 must
  not leave a panic reachable from a config (family check runs first).
- **Drift fixes ride along**: `one --help` says "default: apple1"
  (code: replica1) and "48KB RAM" (code: 32K) — corrected in O3/O2
  respectively rather than left to rot.

## Decisions to make at kickoff

1. **`builtin:list` scope** — all four names everywhere (recommended)
   or per-subcommand filtering.
2. **boo's spelling** in O4 — `--set machine:model=apple1` (minimal) or
   `--config builtin:apple1` (recommended: the builtins then have a
   user besides the CLI).
3. **Bare `ewm one` default** — confirm replica1 (today's code
   behavior; recommended) and fix the usage text, or switch to apple1
   (what the usage text claims).
4. **PR granularity** — per phase, or one PR with phase commits (as
   plan 20260719-01 was executed).

## Backlog (recorded, out of scope)

- **`remote` / `state` for the one family** — REMOTE.md Phase 7 and
  STATE.md; after this plan they are "allow the block for the family
  and wire it", not "build a config surface".
- **Unified launcher** — `ewm --config x.json` picking the subcommand
  from the model family; ties into MAC_APP `.ewmachine` double-click.
- **Display options for `one`** — monitor styles/scanlines for the TTY
  renderer, if ever wanted; the family check is the single place to
  relax.
- **Bootloader menu from builtins** — boo listing every built-in config
  instead of its hardwired three entries.
