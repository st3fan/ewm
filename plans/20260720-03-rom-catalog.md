# A SKU-keyed ROM catalog — consistent `roms/` + one embedded lookup table

- **Design doc:** `notes/ROMS.md` — full inventory of today's five naming
  styles / three extensions / duplicates, the Apple part-number scheme,
  and the rationale. This plan is the execution roadmap; read the note for
  *why*.
- **Status:** in progress — R1 landed. (The original-//e work landed first, so
  the unenhanced `AppleIIe/` ROMs are reorganised once, already wired.)
- **Target:** `main`; one PR per phase.
- **Kickoff decisions (made):** (1) **scope** — all phases R1→R4; (2) **flat**
  `roms/` (dissolve `AppleII/`, `AppleIIe/`; the SKU prefix disambiguates era);
  (3) **SKU-less ROMs** — one catalog keyed by slug (WozMon, Krusader×2,
  apple1-basic); the Klaus test suites stay path-loaded fixtures, not catalog
  entries; (4) **orphans** — deleted (git history preserves them): the
  `.rom.bin` doubles, the keyboard/Signetics/`a2p`/`applesoft-lite`/`341-0868`
  files; the Klaus `.bin`s stay (path-loaded by tests); (5) **one extension**
  `.bin` (the SKU-less `.rom` files converted); (6) `two.rs`/`chr.rs` statics —
  kept as named aliases the catalog backs (R2).

## Goal

Give every ROM one canonical name based on its **Apple part number
(SKU)** — e.g. `342-0304-A` — and make both the `roms/` folder and the
code refer to ROMs through a single lookup table `SKU → (description,
bytes)`. The payoff: `--print-config` shows exactly which silicon a
machine runs, a config can name `"path": "builtin:342-0304-A"`, and the
"what is this ROM" comment lives in one catalog instead of scattered `//`.

Two mechanisms collapse into one. Today Apple ][ / //e machine ROMs are
hard-wired as `include_bytes!` statics in `two.rs` / `chr.rs`, while the
Apple-1-family ROMs go through `config::ROM_BUILTINS` (`builtin:<stem>`).
The catalog is the generalisation of `ROM_BUILTINS` that both use.

## Phases

| Phase | Description | Size | Status |
|---|---|---|---|
| R1 | Canonical `roms/` layout: `git mv` to SKU-first names, one extension, collapse the `341-0036` duplicate, retire `.rom.bin` doubles, dispose of orphans; every `include_bytes!`/`read` path updated in lock-step | M | **Landed** (#TBD) |
| R2 | The catalog: `ROM_CATALOG` of `(key, desc, bytes)` in `rom.rs` owns every machine/char/mouse `include_bytes!`; `two.rs`/`chr.rs`/`mouse.rs` fetch by key; `desc` carries the ROM story | M | **Landed** (#TBD) |
| R3 | `builtin:` resolver reads the catalog so `"path": "builtin:342-0304-A"` resolves; existing slug names keep working; schema regen | S/M | **Landed** (#TBD) |
| R4 *(optional, the payoff)* | Migrate the ][ / ][+ / //e machine ROM wiring out of `two.rs` statics into `configs/*.json` via `builtin:SKU` | L | Not started |

R1 is independent and lands first. R2 → R3 in order. R4 is the invasive
step and may be deferred or split per-machine. Every phase: standard gates
(`cargo fmt --all --check`, `cargo clippy --all-targets -- -D warnings`,
`cargo test` incl. the golden-BMP screenshots) + `readme_examples_parse`.
**The golden-BMP screenshot tests are the tripwire for every phase** — if
a rename or catalog lookup silently swaps the bytes feeding the video
path, they move.

### R1 — canonical `roms/` layout

- **Rename to SKU-first** with `git mv` (history follows): e.g.
  `Apple IIe CD Enhanced - 342-0304-A - 2764.bin` →
  `342-0304-A — Apple IIe CD Enhanced (2764).bin`;
  `3410036.bin` → `341-0036 — Apple II Character ROM (2513).bin`;
  `341-0020.bin` → `341-0020 — Apple II+ Autostart Monitor (2716).bin`.
  SKU-less ROMs (WozMon, Krusader, apple1-basic, Klaus test suites) keep a
  product-name stem — they have no part number.
- **One extension** (`.bin`). Retire the `.rom.bin` Krusader doubles
  (which differ in size from the embedded `.rom` — confirm the `.rom` is
  the intended one first).
- **Collapse the duplicate**: `3410036.bin` ≡ `AppleII/…341-0036.bin`
  (same sha1) — keep one, repoint both references.
- **Flat vs. subfolder** (`AppleII/`, `AppleIIe/`) decided at kickoff; the
  SKU prefix (341 = ][ era, 342 = //e era) already disambiguates.
- **Orphans**: dispose of unreferenced files (`341-0868…`, `a2p.rom`,
  `applesoft-lite.bin`, `Signetics 2513…`, keyboard ROMs, `.rom.bin`
  doubles) — delete (git history preserves them) or keep as provenance,
  per kickoff. **First confirm** `6502_functional_test.bin` /
  `65C02_extended_opcodes_test.bin` aren't loaded by path from a test
  before touching them.
- Update every `include_bytes!` / `std::fs::read(concat!(…))` path in
  `two.rs`, `chr.rs`, `config.rs` in the same commit.
- **Gate:** builds; golden-BMP screenshots byte-identical; existing ROM
  provenance/SHA tests pass; mind `.gitignore`'s `*.bin` +
  `!roms/**/*.bin` negation when adding/renaming.

### R2 — the catalog table

Generalise `config::ROM_BUILTINS` from `(stem, bytes)`:

```rust
struct RomEntry {
    key:  &'static str,   // "342-0304-A", or a slug for SKU-less ROMs
    desc: &'static str,   // "Enhanced //e CD ROM ($C000-$FFFF)"
    data: &'static [u8],  // include_bytes!(…)
}
static ROM_CATALOG: &[RomEntry] = &[ … ];
```

- The catalog owns the `include_bytes!`. `two.rs` / `chr.rs` stop
  declaring `static ROM_341_0011: &[u8] = …` and read
  `catalog("341-0011")` (or keep thin aliases the catalog backs — decide
  at kickoff). The scattered `//` ROM comments move into `desc`.
- **Gate:** builds; golden-BMP unchanged; a test that keys are unique and
  every entry's bytes are non-empty and the expected length.

### R3 — `builtin:SKU` resolver

- `config::rom_builtin` (and the `builtin:` path resolver) read
  `ROM_CATALOG`, so a memory region can name `"path": "builtin:342-0304-A"`.
  Existing slug names (`builtin:WozMon`, `builtin:apple1-basic`,
  `builtin:Krusader-1.3-65C02`) keep resolving — they're catalog keys now.
- Error message lists available keys (as today). Regenerate both schemas
  (`EWM_UPDATE_SCHEMA=1 cargo test -p ewm schema_matches_committed`) if
  the `builtin:` enumeration is reflected there.
- **Gate:** a config that mounts a ROM via `builtin:SKU` loads and boots;
  `builtin:apple1`/`replica1` still boot; `readme_examples_parse`.

### R4 — migrate machine ROM wiring into configs *(optional payoff)*

Move the ][ / ][+ / //e system+video ROM wiring out of the `two.rs` /
`chr.rs` statics into the machine configs (`configs/apple2.json`,
`apple2plus.json`, `apple2enhanced.json`, `apple2e.json`) as
`builtin:SKU` memory regions — the same way `apple1.json` already names
its ROMs. This is where "builtin configs reference ROMs by SKU" fully
lands.

- Biggest blast radius: it changes how machines are constructed
  (`new_2e` / the ][+ builder stop hard-selecting ROMs; the config
  supplies them). May be split per-machine, one PR each.
- **Gate:** golden-BMP screenshots byte-identical for **every** migrated
  machine (the tripwire that the config supplies the exact same bytes);
  each `builtin:` boots to its prompt; `--print-config` shows the SKU
  paths.

## Hazards

- **A missed path is worse than a compile error.** A renamed file with a
  stale-but-still-existing path silently swaps ROM bytes. Renames and path
  edits go in one commit; the golden-BMP screenshots are the tripwire.
- **`git mv`, not delete+add**, so provenance history follows the rename.
- **The `.rom` vs `.rom.bin` Krusader pair differ in size** — confirm the
  embedded `.rom` is authoritative before retiring the `.rom.bin`.
- **`341-0868 …1986-1991…`** is an unrelated Enhanced-era aggregate — do
  not fold it into any machine's ROM set (same caution as the //e plan).
- **Interaction with `20260720-02-original-iie.md`:** that plan adds the
  unenhanced `AppleIIe/` statics + a provenance test. If this reorg runs
  after it lands, R1 renames those too and R2 pulls them into the catalog;
  if it's still in flight, coordinate so the new ROMs are born SKU-keyed
  rather than reorganised twice.

## Decisions to make at kickoff

1. **Scope** — stop at R3 (catalog reachable via `builtin:SKU`) or go
   through R4 (configs actually reference ROMs by SKU). R4 is the full
   vision but the invasive part.
2. **Flat vs. subfolder** `roms/` layout.
3. **SKU-less ROMs** — same catalog keyed by slug, or a second small
   table.
4. **Orphan disposition** — delete vs. keep for provenance (after
   confirming the Klaus test ROMs aren't path-loaded).
5. **`two.rs` statics** — replaced by catalog lookups, or kept as thin
   aliases the catalog backs.
6. **PR granularity** — per phase (default) or fewer; R4 possibly
   per-machine.

## Backlog (recorded, out of scope)

- A `roms/README.md` or generated manifest listing every catalog entry
  (key, desc, size, sha1) — falls out naturally once the catalog exists.
- Verifying every embedded ROM against a published sha1 as a single
  parametrised provenance test over the catalog.
