# A SKU-keyed ROM catalog вАФ consistent `roms/` + one embedded lookup table

- **Design doc:** `notes/ROMS.md` вАФ full inventory of today's five naming
  styles / three extensions / duplicates, the Apple part-number scheme,
  and the rationale. This plan is the execution roadmap; read the note for
  *why*.
- **Status:** complete вАФ R1вАУR5 landed. `roms/` is SKU-first and flat,
  `rom::ROM_CATALOG` is the single source of embedded ROMs, `builtin:<key>`
  reaches any ROM, and the four Apple II configs name their swappable
  motherboard system ROMs (`machine.rom`) and character/video ROM
  (`machine.character_rom`) by SKU вАФ so `--print-config` shows the silicon and
  a chip is swappable. The design that unblocked R4 (owner's, on hardware
  grounds): list the *socketed* motherboard ROMs, not the language-card RAM.
- **Target:** `main`; one PR per phase.
- **Kickoff decisions (made):** (1) **scope** вАФ R1вЖТR3 + R4a landed; R4b (the
  character/video ROM) is the remaining piece. R4 was unblocked by the owner's
  design call: list only the *swappable socketed* motherboard ROMs, not the
  language-card RAM (see the R4 section); (2) **flat**
  `roms/` (dissolve `AppleII/`, `AppleIIe/`; the SKU prefix disambiguates era);
  (3) **SKU-less ROMs** вАФ one catalog keyed by slug (WozMon, Krusader√Ч2,
  apple1-basic); the Klaus test suites stay path-loaded fixtures, not catalog
  entries; (4) **orphans** вАФ deleted (git history preserves them): the
  `.rom.bin` doubles, the keyboard/Signetics/`a2p`/`applesoft-lite`/`341-0868`
  files; the Klaus `.bin`s stay (path-loaded by tests); (5) **one extension**
  `.bin` (the SKU-less `.rom` files converted); (6) `two.rs`/`chr.rs` statics вАФ
  kept as named aliases the catalog backs (R2).

## Goal

Give every ROM one canonical name based on its **Apple part number
(SKU)** вАФ e.g. `342-0304-A` вАФ and make both the `roms/` folder and the
code refer to ROMs through a single lookup table `SKU вЖТ (description,
bytes)`. The payoff: `--print-config` shows exactly which silicon a
machine runs, a config can name `"path": "builtin:342-0304-A"`, and the
"what is this ROM" comment lives in one catalog instead of scattered `//`.

Two mechanisms collapse into one. Today Apple ][ / //e machine ROMs are
hard-wired as `include_bytes!` statics in `two.rs` / `chr.rs`, while the
Apple-1-family ROMs go through `config::ROM_BUILTINS` (`builtin:<stem>`).
The catalog is the generalisation of `ROM_BUILTINS` that both use.

### R1 вАФ canonical `roms/` layout

- **Rename to SKU-first** with `git mv` (history follows): e.g.
  `Apple IIe CD Enhanced - 342-0304-A - 2764.bin` вЖТ
  `342-0304-A вАФ Apple IIe CD Enhanced (2764).bin`;
  `3410036.bin` вЖТ `341-0036 вАФ Apple II Character ROM (2513).bin`;
  `341-0020.bin` вЖТ `341-0020 вАФ Apple II+ Autostart Monitor (2716).bin`.
  SKU-less ROMs (WozMon, Krusader, apple1-basic, Klaus test suites) keep a
  product-name stem вАФ they have no part number.
- **One extension** (`.bin`). Retire the `.rom.bin` Krusader doubles
  (which differ in size from the embedded `.rom` вАФ confirm the `.rom` is
  the intended one first).
- **Collapse the duplicate**: `3410036.bin` вЙ° `AppleII/вА¶341-0036.bin`
  (same sha1) вАФ keep one, repoint both references.
- **Flat vs. subfolder** (`AppleII/`, `AppleIIe/`) decided at kickoff; the
  SKU prefix (341 = ][ era, 342 = //e era) already disambiguates.
- **Orphans**: dispose of unreferenced files (`341-0868вА¶`, `a2p.rom`,
  `applesoft-lite.bin`, `Signetics 2513вА¶`, keyboard ROMs, `.rom.bin`
  doubles) вАФ delete (git history preserves them) or keep as provenance,
  per kickoff. **First confirm** `6502_functional_test.bin` /
  `65C02_extended_opcodes_test.bin` aren't loaded by path from a test
  before touching them.
- Update every `include_bytes!` / `std::fs::read(concat!(вА¶))` path in
  `two.rs`, `chr.rs`, `config.rs` in the same commit.
- **Gate:** builds; golden-BMP screenshots byte-identical; existing ROM
  provenance/SHA tests pass; mind `.gitignore`'s `*.bin` +
  `!roms/**/*.bin` negation when adding/renaming.

### R2 вАФ the catalog table

Generalise `config::ROM_BUILTINS` from `(stem, bytes)`:

```rust
struct RomEntry {
    key:  &'static str,   // "342-0304-A", or a slug for SKU-less ROMs
    desc: &'static str,   // "Enhanced //e CD ROM ($C000-$FFFF)"
    data: &'static [u8],  // include_bytes!(вА¶)
}
static ROM_CATALOG: &[RomEntry] = &[ вА¶ ];
```

- The catalog owns the `include_bytes!`. `two.rs` / `chr.rs` stop
  declaring `static ROM_341_0011: &[u8] = вА¶` and read
  `catalog("341-0011")` (or keep thin aliases the catalog backs вАФ decide
  at kickoff). The scattered `//` ROM comments move into `desc`.
- **Gate:** builds; golden-BMP unchanged; a test that keys are unique and
  every entry's bytes are non-empty and the expected length.

### R3 вАФ `builtin:SKU` resolver

- `config::rom_builtin` (and the `builtin:` path resolver) read
  `ROM_CATALOG`, so a memory region can name `"path": "builtin:342-0304-A"`.
  Existing slug names (`builtin:WozMon`, `builtin:apple1-basic`,
  `builtin:Krusader-1.3-65C02`) keep resolving вАФ they're catalog keys now.
- Error message lists available keys (as today). Regenerate both schemas
  (`EWM_UPDATE_SCHEMA=1 cargo test -p ewm schema_matches_committed`) if
  the `builtin:` enumeration is reflected there.
- **Gate:** a config that mounts a ROM via `builtin:SKU` loads and boots;
  `builtin:apple1`/`replica1` still boot; `readme_examples_parse`.

### R4 вАФ configs reference the socketed ROMs by SKU

**Design (owner's, on hardware grounds):** list only the ROMs a real owner
could *swap* вАФ the socketed **motherboard** chips (`$D000-$FFFF` system ROMs,
the character/video ROM). The language card's `$D000-$FFFF` is 16K of *RAM*,
not a ROM, and its base ROM is the motherboard's вАФ so nothing extra to list,
and no "language card ROM" appears. This resolves the earlier blocker: the
config names the motherboard ROM, and the builder feeds those bytes to
whatever owns the range (the language card `Alc` / the //e `IouE`), so the
banking still works вАФ the config supplies the base ROM instead of a hard-coded
default.

#### R4a вАФ the system ROMs *(landed)*

A `machine.rom` array вАФ `[{ "address": "0xd000", "path": "builtin:341-0011" }, вА¶]`
вАФ names each socketed `$D000-$FFFF` chip (][ / ][+) or the two 8 KB //e halves
(`$C000` CD, `$E000` EF). `config::RomChip` + validation resolve the SKUs;
`build_machine` assembles the chips into the bytes each builder wants
(`new_2plus`/`new_apple2`/`new_2e` gained the ROM as a parameter via
`new_with_slots_and_rom`), and `Two::new` passes `default_rom_chips(model)` so
a bare `ewm two` and its `builtin:` config are the same machine. `IouE`'s
`internal_rom` became owned (was `&'static`) so the //e CD half can come from a
config. The four `configs/apple2*.json` now carry the `rom` array.

- **Gate (met):** the golden-BMP screenshots (the default path is
  byte-identical) **plus** `builtin_configs_build_the_default_system_rom` вАФ
  each config's explicit `machine.rom` builds the same `$D000-$FFFF` as its
  cleared-`rom` default fallback; `schema_matches_committed`;
  `readme_examples_parse`; `--print-config` shows the SKU paths.

#### R4b вАФ the character/video ROM *(landed)*

A `machine.character_rom: "builtin:<SKU>"` names the socketed character (][) /
video (//e) ROM. It feeds the **renderer**, not the builders: the frontend
resolves it and calls `Scr::set_character_rom(rom, model)`, which re-decodes
the model's glyph set (`Chr::from_rom` for the ][, `ChrE::from_rom(_, mousetext)`
for the //e) from that ROM instead of the catalog default. The four
`configs/apple2*.json` carry it (`341-0036`; the //e's `342-0133-A` /
`342-0265-A`), and `--print-config` shows it.

- **Gate (met):** `set_character_rom_decodes_from_the_given_rom` вАФ re-applying
  a model's default video ROM is byte-identical, and a swapped ROM re-decodes
  the glyphs (the Enhanced-vs-original MouseText slot moves). The golden-BMP
  screenshots (default path) stay byte-identical; `schema_matches_committed`.

**As built (R4):** the whole plan (R1вАУR4) is done вАФ `roms/` is SKU-first and
flat, `rom::ROM_CATALOG` is the single embedded-ROM source, `builtin:<key>`
reaches any ROM, and the Apple II configs name their swappable motherboard
system + character ROMs by SKU (the language-card RAM, correctly, is not a
ROM and is not listed).

### R5 вАФ commented config JSON

Config parsing is now lenient (JSONC): a hand-rolled `strip_json_comments`
removes `//` line and `/* */` block comments (leaving string contents and
line numbers intact) before `serde_json`, applied at the two parse points
everything funnels through (`load_builtin`, `from_str_partial`). No new crate
вАФ comments are the only leniency (not trailing commas). Every built-in config
now annotates each `builtin:<key>` ROM with its catalog description (via
`rom::describe`), so the ROM listings read as names вАФ e.g.
`{ вА¶"builtin:341-0011" }, // Apple ][+ AppleSoft BASIC, $D000-$D7FF`.

- **Gate (met):** `strips_comments_but_not_strings` (comments go, string
  contents stay); `every_config_rom_line_has_its_catalog_name_as_a_comment`
  (the annotations match the catalog and can't drift); the existing
  builtin-config load/round-trip tests pass against the now-commented files.

## Hazards

- **A missed path is worse than a compile error.** A renamed file with a
  stale-but-still-existing path silently swaps ROM bytes. Renames and path
  edits go in one commit; the golden-BMP screenshots are the tripwire.
- **`git mv`, not delete+add**, so provenance history follows the rename.
- **The `.rom` vs `.rom.bin` Krusader pair differ in size** вАФ confirm the
  embedded `.rom` is authoritative before retiring the `.rom.bin`.
- **`341-0868 вА¶1986-1991вА¶`** is an unrelated Enhanced-era aggregate вАФ do
  not fold it into any machine's ROM set (same caution as the //e plan).
- **Interaction with `20260720-02-original-iie.md`:** that plan adds the
  unenhanced `AppleIIe/` statics + a provenance test. If this reorg runs
  after it lands, R1 renames those too and R2 pulls them into the catalog;
  if it's still in flight, coordinate so the new ROMs are born SKU-keyed
  rather than reorganised twice.

## Decisions to make at kickoff

1. **Scope** вАФ stop at R3 (catalog reachable via `builtin:SKU`) or go
   through R4 (configs actually reference ROMs by SKU). R4 is the full
   vision but the invasive part.
2. **Flat vs. subfolder** `roms/` layout.
3. **SKU-less ROMs** вАФ same catalog keyed by slug, or a second small
   table.
4. **Orphan disposition** вАФ delete vs. keep for provenance (after
   confirming the Klaus test ROMs aren't path-loaded).
5. **`two.rs` statics** вАФ replaced by catalog lookups, or kept as thin
   aliases the catalog backs.
6. **PR granularity** вАФ per phase (default) or fewer; R4 possibly
   per-machine.
