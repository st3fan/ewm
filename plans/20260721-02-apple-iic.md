# The Apple //c (`model: apple2c`, `builtin:apple2c`)

- **Design docs:** this plan carries its own design (what a //c adds over
  the Enhanced //e, the board model, the soft-switch and serial deltas) —
  self-contained like the original-][ and telnet plans; the durable
  memory-map/IOU forensics become `notes/APPLE_IIC.md` as an as-built
  (Phase C4), the way `notes/APPLE_IIE_ENHANCED.md` is the //e's. Read
  first: `notes/APPLE_IIE_ENHANCED.md` (the machinery this reuses —
  65C02, MMU/IOU, aux memory, 80-col, DHGR, the enhanced char ROM),
  `notes/JSON_CONFIG.md` (the model/family/validation model this
  extends), and — for the mouse — the in-flight AppleMouse plan
  (`plans/20260721-01-apple-mouse-card.md`). Primary sources: the *Apple
  //c Technical Reference Manual* and the *Apple //c ROM (version 255)*
  firmware.
- **Backlog origin:** `notes/IDEAS.md` → "Apple //c (L) — MMU/IOU cousins
  of the //e work, built-in drives, mouse, serial. The //e plan's future
  work"; and `APPLE_IIE_ENHANCED.md` "Future work".
- **Target this pass (owner's kickoff answers):** the **original //c
  (ROM version 255)** — 16K ROM, one built-in 5.25″ drive, fixed 128K,
  two serial ports, mouse; **no** `$C028` ROM bank switch (a ROM 3+
  feature). Serial ports land as a **boot-satisfying 6551 stub**, not the
  full modem bridge. Both are backlog items below.
- **Status:** planned — not started. Gated on the //c ROM (see Hazards)
  and on the AppleMouse card landing (C3 depends on it).
- **Target branch:** `main`; PR granularity decided at kickoff.

## Why this is mostly //e reuse

A //c is an Enhanced //e in a sealed case: same 65C02, same MMU/IOU
soft-switch model, same 128K aux banking, same 80-column / DLGR / DHGR
video, same enhanced character ROM (lower case + MouseText). All of that
is already built and gated for the //e (`IouE`, `new_2e`, the 560-wide
renderer, `aux/`). The //c-specific work is a short list:

1. **The //c ROM** (its peripheral firmware is *in the ROM*, not on cards).
2. **Two 6551 serial ACIAs** — a device class EWM does not have at all.
3. **A //c IOU variant** — a handful of switches the //e deliberately
   makes inert are *live* on the //c.
4. **A fixed-board config model** — the //c has no user-populatable slots;
   its cards are soldered in.
5. **Wiring the built-in mouse and 5.25″ drive** as that fixed board
   (firmware-in-ROM, IOU-driven), reusing the AppleMouse device and `Dsk`.

## The machine (original //c, ROM 255)

- **CPU/RAM:** 65C02, 128K (main + aux), always — the //c has no 48K/64K
  build to configure. `new_2c` mirrors `new_2e` but fixes the aux at 128K
  (the extended-80-col path) and needs no aux-card selection.
- **ROM:** the 16K //c system ROM at `$C100-$FFFF` (with the `$C800`
  expansion region internal, as the //e's `$CX` internal ROM already is —
  the //c has no external cards contending for it, which *simplifies* the
  `$C800` arbitration `two.rs` already implements for the //e). No
  `$C028` bank switch on ROM 255.
- **Character / video ROM:** the //c's is the MouseText-bearing enhanced
  set. **Reuse the committed enhanced //e char ROM if byte-identical**
  (assert it with a hash, exactly as `original-apple2` reused
  `3410036.bin`); embed the //c's only if it differs.
- **IOU deltas (`IouC`, an `IouE` cousin):**
  - **IOUDIS is real on the //c.** `$C07E` (IOUDIS on) / `$C07F` (off),
    with `RDIOUDIS`/`RDDHIRES` reads — the switch EWM made **inert** for
    the //e on purpose (`APPLE_IIE_ENHANCED.md` quirk 8: "//c behavior
    copy-pasted in error"). On the //c it gates DHIRES *and* the
    mouse/VBL interrupt enables. This must be modeled **without touching
    the //e path** (the //e's inert `$C07E/$C07F` is correct and
    golden-tested).
  - **Mouse / VBL interrupt switches** in the IOU (X0/Y0 movement, button,
    VBL-interrupt enable) — the //c drives its mouse through these, not a
    slot-4 card ROM. This is where the real forensics live; the exact map
    comes from the //c Technical Reference in C0/C1.
  - **The 40/80 column switch** and the built-in keyboard (Open/Solid
    Apple, keyboard-mouse option) — mostly //e-compatible.

## The serial ports (boot-satisfying 6551 stub)

The //c's two ports are 6551 ACIAs in the phantom slot-1 / slot-2 DEVSEL
ranges (Port 1/printer ≈ `$C098-$C09B`, Port 2/modem ≈ `$C0A8-$C0AB` —
confirm against the Technical Reference in C2). EWM has **no ACIA code**.

The stub is the minimum for the firmware to initialize and probe both
ports without wedging: the **status register reads back "transmit
register empty, receiver empty"** so the firmware's TX-ready wait
completes and its RX poll returns nothing; control/command register
writes are accepted and stored; transmitted bytes are discarded; no data
arrives. Enough that boot, `PR#1`, and `IN#2` return cleanly. The full
6551 (real TX/RX bridged to a TCP socket — the Super Serial Card /
virtual-modem idea in `IDEAS.md`) is a backlog item.

## The fixed-board config model

The //c is soldered: no user `slots`, no aux-card choice. In
`validate_complete`, `apple2c` **rejects `machine.aux`** (128K is built
in) and **constrains `machine.slots`** — the board is fixed, so the only
thing a document may set is the **built-in 5.25″ drive's image**. The
sanctioned form reuses the existing disk plumbing:

```json
{ "machine": { "model": "apple2c",
    "slots": { "6": { "card": "diskii", "drive1": "disks/DOS33.dsk" } } } }
```

— i.e. slot 6 may carry *only* the built-in drive's image (drive 2 and
any other slot/card assignment rejected), everything else being fixed
hardware. (Kickoff alternative: a dedicated `machine.disk` key instead of
a constrained `slots`; recommendation is the constrained `slots` — it
reuses `SlotCard::Diskii`, the fetch/self-containment sweep, and
`--print-config` round-trip untouched.)

## Phases

| Phase | Description | Size | Status |
|---|---|---|---|
| C0 | //c ROM assets + the //c memory/soft-switch/IOU map documented (provenance + hash test; char-ROM reuse) | S/M | Planned |
| C1 | `model: apple2c` + `new_2c` + `IouC` (live IOUDIS/DHIRES, //c switches); boots headless to `]` in 80 columns | M/L | Planned |
| C2 | The 6551 serial stub as a device in the phantom slot-1/2 ranges; firmware init/probe no longer wedges | M | Planned |
| C3 | The fixed board: reject slots/aux, wire the built-in mouse (from the AppleMouse card) + built-in 5.25″ drive; `builtin:apple2c` boots a disk | M/L | Planned |
| C4 | Docs + as-built `notes/APPLE_IIC.md`; README `apple2c` profile; schema inventory; tick `IDEAS.md` | S | Planned |

Order **C0 → C1 → C2 → C3 → C4**, with a wrinkle: the //c firmware may
hang on serial init *before* reaching `]`, so C1 either carries a throwaway
inline serial no-op to reach the prompt or is landed together with C2 —
decide once the ROM is in hand (a boot trace shows whether it wedges).
C3's mouse wiring **depends on the AppleMouse card** (`20260721-01`)
having landed its device + the shared IRQ line. Every phase runs the
standard gates (`cargo fmt --all --check`, `cargo clippy --all-targets --
-D warnings`, full `cargo test` **including the golden-BMP screenshots** —
the tripwire that the //c work left the ][+/ //e renderers untouched)
plus the phase gate.

### C0 — ROM assets + the map

- Embed the owner-supplied original //c ROM (version 255) following the
  current ROM-asset convention — the `include_bytes!` static in `two.rs`
  today, or `builtin:SKU` if the ROM-catalog plan (`20260720-03`) has
  landed by then (fold into whichever exists; don't block on the catalog).
- A SHA-1 provenance/hash test pins the image (the crate's `ws::sha1`, as
  the //e and AppleII ROMs are pinned); assert the //c char ROM equals the
  committed enhanced //e char ROM if byte-identical, so the reuse can't
  drift.
- Document the //c memory map and the //c-only soft switches (IOUDIS,
  mouse/VBL interrupts, serial addresses) — the seed of `notes/APPLE_IIC.md`.
- **Gate:** the hash/provenance test; the map documented. (Statics unused
  until C1 → `#[allow(dead_code)]` or land C0+C1 together.)

### C1 — The `apple2c` machine + `IouC`

- `config::Model::TwoC` (token `"apple2c"`, family `Apple2`,
  `two_type → TwoType::Apple2C`); the `TwoType::Apple2C` arm in
  `build_machine` becomes the real `new_2c` constructor.
- `new_2c` mirrors `new_2e`: 65C02, //c ROM at `$C100-$FFFF`, fixed 128K
  aux, the enhanced renderer — but an `IouC` that makes **IOUDIS /
  `$C07E` / `$C07F` / RDIOUDIS / RDDHIRES live** and adds the //c
  mouse/VBL interrupt + 40/80 switches. Share `IouE` code where the
  behavior is identical; the //e path must stay byte-for-byte unchanged
  (its inert `$C07E/$C07F` is deliberate).
- **Gate:** a `two_c_boot.rs` (mirroring `two_e_boot.rs`): reset lands at
  the BASIC `]` prompt with 80-column firmware active; a switch test pins
  the //c IOUDIS/DHIRES truth table (distinct from the //e's); the //e and
  ][+ boot/golden suites are untouched.

### C2 — The 6551 serial stub

- A `Mos6551` (`Device` + `Persist`) mapped into the phantom slot-1 and
  slot-2 DEVSEL ranges at the ACIA register offsets; status returns
  TX-empty / RX-empty, command/control writes stored, TX discarded, RX
  empty (the stub contract above).
- **Gate:** a headless test that the //c firmware's port init reads back
  sane status and `PR#1` / `IN#2` return to BASIC without hanging;
  boot-to-`]` is clean whether or not C1 needed the throwaway stub.

### C3 — The fixed board: mouse + built-in drive

- `validate_complete`: `apple2c` rejects `machine.aux` and constrains
  `machine.slots` to only the built-in drive's image at slot 6 (per the
  board model above); a new family/model row in the validation table.
- Wire the **built-in mouse** — reuse the AppleMouse `Mou` device and its
  coordinate/interrupt model, but driven by `IouC`'s mouse/VBL switches
  and the //c ROM's firmware (no slot-4 card ROM), asserting the shared
  IRQ line from the mouse plan's M1.
- Wire the **built-in 5.25″ drive** — reuse `Dsk` at the phantom slot-6
  IWM range; verify the //c ROM boots DOS 3.3 / ProDOS from it.
- `configs/apple2c.json` + `builtin:apple2c`; `builtin:list` and the
  name/self-containment tests extend to it.
- **Gate:** `builtin:apple2c` boots a DOS 3.3 (or ProDOS) disk from the
  built-in drive; the mouse is detectable at the phantom slot 4 and tracks
  under the //c firmware (a scripted end-to-end, matching the mouse plan's
  M4 style).

### C4 — Docs + as-built note

- `notes/APPLE_IIC.md`: the //c memory map, the `IouC` deltas (live

- README: an `apple2c` `two` profile (checked by `readme_examples_parse`)

## Hazards

- **ROM sourcing is the blocker** — the owner's call, same provenance
  stance as the committed //e / AppleII ROMs. `roms/` has no //c ROM
  today; there is one unidentified 32K `341-0868 … W5` image (unreferenced,
  no Apple II machine strings — likely a WIZnet/Uthernet asset, *not* the
  //c ROM). **Confirm the //c ROM 255 image before C0.**
- **The //c firmware drives built-ins we must service, or boot wedges.**
  Serial init, mouse/VBL, and disk probes all run at boot; a probe that
  never satisfies hangs before `]`. Sequence so the machine reaches the
  prompt early with minimal stubs, then flesh out — and take a boot trace
  first thing in C1.
- **IOUDIS is model-dependent.** The //e's inert `$C07E/$C07F` is correct
  and golden-tested; making them live for the //c must not regress the //e.
  Parameterize the IOU by model; do not flip a shared flag.
- **Fixed-board config.** `apple2c` must reject arbitrary slots/aux and
  yet still accept a boot disk — the constrained-`slots` model above is the
  seam; a slip lets a //c express hardware it cannot have.
- **//c disk is IWM, not a Disk ][ card.** `Dsk`'s soft-switch decode
  ($C0E0-$C0EF) should match closely, but the //c ROM's disk firmware is
  the real test — verify a boot rather than assuming reuse.
- **Cross-thread dependency.** C3 needs the AppleMouse device + IRQ line;
  if that plan slips, C3 slips (C0-C2 do not).
- **Determinism / golden-BMP.** New //c goldens; the ][+ and //e goldens
  must not move — the standing tripwire.

## Decisions to make at kickoff

1. **The //c ROM image** — confirm the owner-supplied ROM 255 bytes (and
   whether `341-0868 … W5` is relevant, expected: no).
2. **Char/video ROM** — reuse the enhanced //e char ROM (recommended, if
   byte-identical, with a hash assertion) or embed the //c's.
3. **Boot disk config surface** — constrained `machine.slots` at slot 6
   (recommended; reuses `SlotCard::Diskii`) vs a dedicated `machine.disk`
   key.
4. **C1/C2 ordering** — land C1 with a throwaway serial no-op, or land
   C1+C2 together (decide from the boot trace).
5. **As-built note timing** — write `notes/APPLE_IIC.md` up front as a
   design note, or produce it as the C4 as-built (recommended, per the //e
   precedent).
6. **PR granularity** — one PR per phase (default) or fewer.

## Backlog (recorded, out of scope)

- **//c ROM 3/4 + memory expansion** — the `$C028` bank-switched 32K ROM,
  3.5″ / RAM-disk support, the memory-expansion card. A second profile
  once ROM 255 lands.
- **Apple //c Plus** — 4 MHz accelerator + built-in 3.5″ (IWM 3.5). Both
  the accelerator and the 3.5″ drive are new; the biggest departure.
- **Full 6551 + virtual modem** — real TX/RX bridged to a TCP socket for
  period terminal software to "dial" telnet BBSes (`IDEAS.md`); also
  lands the standalone **Super Serial Card** for the ][+ / //e.
- **//c self-test / diagnostics** (Open-Apple + Ctrl + Reset) and the
  **keyboard-mouse** option — fidelity polish.
- **NMOS //e** (`20260720-02`) and the **ROM catalog** (`20260720-03`) are
  sibling efforts this plan coordinates with but does not depend on.
